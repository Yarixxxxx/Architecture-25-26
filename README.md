# Лабораторная работа №2

## Вариант 15: 
Крупная компания разрабатывает систему бронирования и управления отелями премиум-класса.

## Тема: 
Использование нотации C4 model для проектирования архитектуры программной системы

## Цель работы: 
Получить опыт использования графической нотации для фиксации архитектурных решений.

### Акторы:

Гость - основной потребитель услуг системы, использует систему для бронирования номеров и управления доступом

Администратор отеля - сотрудник, ответственный за управление бронированиями и мониторинг состояния номеров

Персонал по уборке - технический персонал, получающий задания на обслуживание номеров

### Целевая система:

Система бронирования отелей - разрабатываемая программная система, которая является центральным элементом диаграммы

### Внешние системы:

Существующая система бронирования - унаследованная система для обработки платежей и хранения данных гостей

Система умных замков - оборудование для управления доступом в номера

Email сервис - внешний сервис для отправки уведомлений


## Диаграмма контейнеров

<img width="1058" height="967" alt="image" src="https://github.com/user-attachments/assets/36e40ad3-d98a-4b3d-8a4d-ea1d1675a644" />

### Контейнеры:

Веб-приложение - одностраничное приложение для администраторов и персонала

Мобильное приложение - кросс-платформенное приложение для гостей отеля

Бэкенд API - серверная часть, реализующая бизнес-логику и REST API

База данных - реляционная БД для хранения структурированных данных

Redis - in-memory хранилище для кэширования и сессий

Внешние системы - те же, что и на диаграмме контекста



#### Выбрана многоуровневая клиент-серверная архитектура с элементами микросервисного подхода для интеграций.



### Причины выбора:

### 1. Разделение ответственности:

- Фронтенд отвечает за представление данных

- Бэкенд API реализует бизнес-логику и безопасность

- База данных обеспечивает надежное хранение данных

### 2. Независимость развертывания:

- Каждый контейнер может развертываться и масштабироваться независимо

- Возможность использовать разные технологии для разных компонентов

### 3. Гибкость интеграции:

- REST API обеспечивает слабую связность между компонентами

- Возможность замены отдельных компонентов без переписывания всей системы

### 4. Масштабируемость:

- Статический фронтенд может развертываться на CDN

- Бэкенд API и база данных могут масштабироваться горизонтально

- Redis повышает производительность за счет кэширования

### 5. Безопасность:

- Единая точка входа для контроля доступа

- Возможность централизованного аудита и мониторинга

## Диаграмма компонентов - Backend API

<img width="1854" height="1074" alt="image" src="https://github.com/user-attachments/assets/8ad86331-81bc-47ea-83d1-5fb6afb15ec6" />

### API Gateway 
Единая точка входа для всех API запросов, обеспечивающая маршрутизацию и предварительную аутентификацию

### Контроллеры:

Booking Controller - обработка запросов, связанных с бронированием

Auth Controller - управление аутентификацией и авторизацией

Room Controller - операции с номерами отеля

Cleaning Controller - управление заданиями по уборке

### Сервисы бизнес-логики:

Booking Service - основная логика бронирования

Auth Service - управление пользователями и сессиями

Room Service - управление состоянием и доступностью номеров

Cleaning Service - алгоритмы приоритизации уборки

Notification Service - отправка уведомлений пользователям

### Интеграционные компоненты:

Payment Integration - адаптер для работы с платежной системой

Lock Integration - адаптер для управления умными замками

### Репозитории доступа к данным:

Booking Repository - доступ к данным о бронированиях

User Repository - доступ к данным пользователей

Room Repository - доступ к данным о номерах


## Диаграмма компонентов - Web Application

<img width="2162" height="1016" alt="image" src="https://github.com/user-attachments/assets/a5553280-4fb5-47fa-9f89-25c170c512f5" />

### Router 
Управление навигацией между различными модулями приложения

### Функциональные модули:

Auth Module - компоненты для входа, регистрации и управления профилем

Booking Module - интерфейс для работы с бронированиями

Room Module - управление номерами и их статусами

Cleaning Module - специализированный интерфейс для персонала уборки

Reporting Module - формирование отчетов и аналитика

### Сервисные компоненты:

API Client - унифицированный клиент для REST API вызовов

State Manager - управление глобальным состоянием приложения

UI Components - библиотека переиспользуемых компонентов интерфейса

WebSocket Client - поддержка реального времени для обновления статусов

### Хранилища состояния:

Auth Store - хранение состояния аутентификации и пользовательских данных

Booking Store - кэширование данных о бронированиях

Room Store - хранение актуального состояния номеров
